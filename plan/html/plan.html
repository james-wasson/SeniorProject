  1. Introduction

This document will outline the work to be completed to create the website Inventor Access Portal for the Office of Technology Transfer (OTT). This document will also include what work each individual shall complete and be held responsible for. The documentation here should be a complete list of the work to be completed within and for the project to be considered acceptable.

This document is held as a semi-flexible plan as such all other work requests or added functionality must be discussed, agreed upon by all parties involved, and added to this document before the request can be completed.

  1. Statement of Work

The OTT has requested the development of a web portal that allows inventors from the Southern Illinois University system to access information regarding technologies they have invented. This would reduce the number of calls the OTT receives from said inventors by consolidating inventor information in an easy to access portal that can be accessed through the internet.

The final product will consist of a front-end portal from which the inventors can log-in with their credentials and view information about projects and patents they are a part of or were a part of. This will be hosted on a server which will communicate information from and to the database containing all of the inventor information.

The system that the OTT currently uses utilizes an Access database, therefore the data retrieval tools will be largely Access based.

  1. Acceptance Criteria

The work will be deemed acceptable once it meets the following functional requirements:

  • Ability to log-in to the system
  • Ability to view projects
  • Ability to sort projects
  • Ability to view different reports
  • On login a confidentiality notice is displayed
  • Ability for the user to change password
  • Ability for the user to change their email
  • Ability for the user to change their username
  • Ability to submit a new user form
  • Recover a lost password
  • Ability to view project reports
  • Print any reports generated by the website
  • Keep a log of emails
  • Keep a log of signins

The work must also meet the following non-functional requirements:

  • Database access must be secure
  • The database will be modified to include login fields
  • Login will be secured
  • Look and feel should match parent website
  • Allows users to view the family of projects that the user is a member of
  • The bottom of each page should display a confidentiality notice

The following technical requirements must also be met for the work to be deemed acceptable:

  • System will be regularly available
  • The system will work quickly
  • The system will have an interface to access the current Access database

The following documents/items should be able to be delivered:

  • Code fulfilling all of the above requirements
  • Documentation

  1. Resource Requirements

Test environment:

Server

The Inventor Access Portal test environment will consist of a host server running Apache2, C#, and ACCESS Database Management System. A public IP address or domain name may be needed for remote access.

Client

To ensure that The Inventor Access Portal can be accessed by most of the browsers in different devices (mostly Android, IOS, Mac, and Windows.), different browsers in different platform environment is needed as clients for test.

  1. Risks

Risk

Impact

Action

Lack of communication between team members

Any lack of communication could lead to miscommunication which can hamper progress in completing what needs to be done.

Knowing ways to contact each other and responding to each other regularly can help avoid this issue.

Unfamiliarity with languages and technology being used

This could lead to team members lacking expertise with the languages and technology being used to have to play catch-up which could eat into time that could be spent more productively.

Team members should conduct outside research to quickly familiarize themselves to be able to use the languages and technology comfortably.

Time Management

Inability to meet deadlines the group sets and not spending enough time on the project where possible can put the group behind where they need to be.

Constant reminders of what needs to be done and by when to keep team members aware could help, but ultimately it’s up to the individual team members to manage effective project time.

Lack of communication with mentor

Between devoting a lot of time working on the project itself along with other deliverables, it can become easy to forget to update our mentor about our progress, not consulting them could potentially lead to diversions that aren’t relevant to the final desired product that was requested.

Frequent communications with questions as well as updates with our mentor can easily mitigate this potential risk.

Unable to route backend functions to frontend portals.

Sometimes, problems will occur while trying to route backend functions to frontend portals due to the difference in structure design or different numbers of parameters for example.

Design the frontend and backend carefully and reduce the conflicts between them as much as possible. Check the whole structure again after the design of both frontend and backend.  

Unable to connect Access database with C# .net technology

If Access database is not supported by .net technology, our previous design will not work and we have to redesign the database to a Mysql version and use Mysql database management system to meet the requirement instead.

Do research on data transfer methods between Access database and .net and do some test ahead of time.

Limitation of platform

Since we used MS-Access as our database, our server can only be Windows server. If server system is changed, a lot of adjustment need to be made to make the project work.

Plan it ahead of time.
